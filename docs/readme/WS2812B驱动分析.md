# WS2812B 驱动原理

> 目标：这篇文档只分析 WS2812B 如何被驱动，不绑定某个具体项目、芯片平台或代码实现。重点从数据流角度理解：应用层的 RGB 颜色，如何一步步变成 WS2812B 能识别的单线时序波形。
>
> 适合记住的核心结论：**WS2812B 不是通过寄存器配置颜色，而是通过 DIN 数据线上一串严格定时的高低电平脉冲来接收颜色数据。**

## 1. 先用一句话理解 WS2812B

**WS2812B 是一颗内置控制 IC 的 RGB LED，它通过一根数据线 `DIN` 接收颜色数据，每颗灯通常接收 24 bit：8 bit Green、8 bit Red、8 bit Blue。**

它的控制方式和 I2C、SPI、UART 这类常见通信不同：

- 没有设备地址。
- 没有寄存器读写。
- 没有单独时钟线。
- 没有 ACK 应答。
- 只有一根数据输入线 `DIN`，靠高低电平持续时间表示 `0` 和 `1`。

可以先按下面的结构理解：

```text
-------------+      DIN       +-----------+ DOUT  +-----------+ DOUT
| MCU / SoC  | -------------->| WS2812B 1 | ----> | WS2812B 2 | ----> ...
+-------------+                +-----------+       +-----------+
```

主控只需要连续输出一串符合时序要求的波形。第一颗灯吃掉最前面的 24 bit，剩余数据从 `DOUT` 传给下一颗灯，后面的灯依次类推。

## 2. WS2812B 需要什么数据

从应用层看，我们通常想设置的是 RGB：

```text
R = 255
G = 0
B = 0
```

但 WS2812B 常见的数据顺序不是 `RGB`，而是 `GRB`：

```text
G7 G6 G5 G4 G3 G2 G1 G0
R7 R6 R5 R4 R3 R2 R1 R0
B7 B6 B5 B4 B3 B2 B1 B0
```

也就是说，一颗灯的数据是：

```text
1 颗 WS2812B = 8 bit Green + 8 bit Red + 8 bit Blue = 24 bit
```

如果有 N 颗灯，主控要连续发送：

```text
第 1 颗灯的 24 bit
+ 第 2 颗灯的 24 bit
+ 第 3 颗灯的 24 bit
+ ...
+ 第 N 颗灯的 24 bit
```

数据流可以表示为：

```text
应用层颜色
  RGB, RGB, RGB ...
      |
      v
按 WS2812B 要求重排
  GRB, GRB, GRB ...
      |
      v
按 bit 从高位到低位展开
  G7...G0 R7...R0 B7...B0
      |
      v
转换成单线高低电平脉冲
```

## 3. 底层原理：0 和 1 由脉冲宽度表示

WS2812B 判断一个 bit 是 `0` 还是 `1`，看的不是“高电平就是 1，低电平就是 0”，而是一个 bit 周期内 **高电平持续多久、低电平持续多久**。

常见时序可以近似理解为：

| bit 值 | 高电平时间 | 低电平时间 | 含义 |
| --- | --- | --- | --- |
| `0` | 短 | 长 | 高电平占空比较小 |
| `1` | 长 | 短 | 高电平占空比较大 |

用波形表示：

```text
bit 0:  ┌──┐________
        │  │
________┘  └________
        高短  低长

bit 1:  ┌──────┐____
        │      │
________┘      └____
        高长    低短
```

更直白地说：

- `0`：先拉高一小段时间，再拉低一长段时间。
- `1`：先拉高一长段时间，再拉低一小段时间。

所以 WS2812B 驱动的本质不是“发送普通字节”，而是：

```text
把每一个颜色 bit 翻译成一个严格定时的高低电平脉冲。
```

## 4. 完整数据流

以显示红色为例，应用层可能写的是：

```text
R = 255 = 0xFF
G = 0   = 0x00
B = 0   = 0x00
```

但真正送到 WS2812B 的顺序是 `GRB`：

```text
G = 0x00
R = 0xFF
B = 0x00
```

展开成 bit：

```text
G: 00000000
R: 11111111
B: 00000000
```

最终数据流是：

```text
应用层 RGB
  R=0xFF, G=0x00, B=0x00
        |
        v
重排为 GRB
  G=0x00, R=0xFF, B=0x00
        |
        v
展开为 24 bit
  00000000 11111111 00000000
        |
        v
bit 0 变成“高短低长”
bit 1 变成“高长低短”
        |
        v
DIN 上输出连续脉冲
        |
        v
WS2812B 解码并暂存颜色
        |
        v
reset 低电平后锁存显示
```

注意：在发送 24 bit 的过程中，WS2812B 只是接收和暂存数据。通常要等到数据线保持低电平超过一段 reset 时间后，灯珠才会把刚收到的数据锁存到内部 PWM 输出，颜色才稳定显示出来。

## 5. 多颗灯级联时数据怎么流动

WS2812B 很适合做灯带，因为每颗灯都有 `DIN` 和 `DOUT`：

```text
MCU
 |
 v
+--------+   +--------+   +--------+
| LED 1  |-->| LED 2  |-->| LED 3  |--> ...
+--------+   +--------+   +--------+
```

主控发送的数据不是“带地址的数据包”，而是一整串连续颜色数据：

```text
LED1 的 GRB 24 bit
LED2 的 GRB 24 bit
LED3 的 GRB 24 bit
...
```

每颗灯的处理方式可以理解为：

1. 从 `DIN` 接收数据流。
2. 消费最前面的 24 bit，作为自己的颜色。
3. 把后面的数据整形后从 `DOUT` 继续输出。
4. reset 低电平到来后，锁存自己的 24 bit 并显示。

所以第几颗灯显示什么颜色，取决于它在整条数据流中的位置：

```text
发送顺序：
[LED1颜色][LED2颜色][LED3颜色][LED4颜色]

实际结果：
第 1 颗灯拿第 1 组 24 bit
第 2 颗灯拿第 2 组 24 bit
第 3 颗灯拿第 3 组 24 bit
第 4 颗灯拿第 4 组 24 bit
```

这也是为什么更新灯带时通常要重新发送整条灯带的数据。因为 WS2812B 没有地址，也不能单独告诉“第 10 颗灯变红”；主控要构造完整帧，让每颗灯按顺序拿到属于自己的 24 bit。

## 6. reset 低电平的作用

发送完所有灯珠的颜色数据后，主控需要让数据线保持低电平一段时间。这个低电平通常称为 reset 或 latch。

它的作用是：

- 告诉 WS2812B：这一帧数据已经结束。
- 让每颗灯把刚接收的 24 bit 锁存到内部寄存器。
- 内部 PWM 根据锁存值开始输出对应亮度。

可以把一次完整刷新理解为：

```text
连续发送 N 颗灯的颜色脉冲
        |
        v
保持低电平 reset
        |
        v
所有灯同时锁存并显示新颜色
```

如果 reset 低电平不够，可能出现颜色不刷新、数据串帧、显示错乱等问题。

## 7. 亮度是如何体现的

WS2812B 每个颜色通道 8 bit，所以单个通道的取值范围通常是：

```text
0 ~ 255
```

这个数值不是直接改变外部数据线的电压，而是写入灯珠内部后，用来控制内部 PWM 占空比。

例如：

```text
R = 255  -> 红色通道最亮
R = 128  -> 红色通道约半亮
R = 0    -> 红色通道关闭
```

从数据流角度看，亮度只是颜色字节的数值不同：

```text
更亮的红色：R = 11111111
较暗的红色：R = 00100000
关闭红色：  R = 00000000
```

主控发送完这些 bit 后，WS2812B 内部再把它们转成 PWM 亮度。

所以整体链路是：

```text
颜色数值
  -> 8 bit 通道值
  -> GRB 24 bit 数据
  -> DIN 定时脉冲
  -> 内部锁存
  -> 内部 PWM 控制 LED 亮度
```

## 8. ESP32-S3 如何产生 WS2812B 波形

ESP32-S3 上可以用软件精确延时、RMT 脉冲外设或 SPI 外设驱动 WS2812B。三种方式只是“波形生成器”不同，上层的 `RGB -> GRB -> 24 bit`数据流不变。

WS2812B 常见的单 bit 周期约为 `1.25 us`，下表数值仅用于理解和伪代码，实际工程应以所用灯珠批次的数据手册容差为准：

| 符号 | 高电平 | 低电平 | 总时间 |
| --- | ---: | ---: | ---: |
| `0` | 约 `0.4 us` | 约 `0.85 us` | 约 `1.25 us` |
| `1` | 约 `0.8 us` | 约 `0.45 us` | 约 `1.25 us` |
| reset/latch | - | 保持低电平，通常留 `>= 80 us` 余量 | - |

### 8.1 定时器或精确延时驱动（不建议，仅作引导）

这种方式由 ESP32-S3 CPU 直接翻转 GPIO。由于一个 bit 只有约 `1.25 us`，普通的毫秒级 FreeRTOS 延时不能使用；即使微秒级通用延时函数也可能粒度不够，通常要用 CPU cycle counter、短循环或定时器计数值忙等。

发送期间还要防止中断和任务切换把某个高电平拉长。下面是表达原理的伪代码：

```c
typedef struct {
    uint8_t r;
    uint8_t g;
    uint8_t b;
} pixel_t;

static void wait_cycles(uint32_t cycles)
{
    uint32_t start = cpu_cycle_count();

    while ((uint32_t)(cpu_cycle_count() - start) < cycles) {
        ;                           // 忙等，不调度
    }
}

static void send_ws2812_bit(uint8_t bit)
{
    gpio_set_high();
    if (bit == 0U) {
        wait_cycles(CYCLES_FOR_0_4_US);
        gpio_set_low();
        wait_cycles(CYCLES_FOR_0_85_US);
    } else {
        wait_cycles(CYCLES_FOR_0_8_US);
        gpio_set_low();
        wait_cycles(CYCLES_FOR_0_45_US);
    }
}

static void ws2812_show(const pixel_t *pixels, size_t led_count)
{
    enter_timing_critical_section(); // 伪代码：在可控时间内禁止抢占/干扰

    for (size_t i = 0; i < led_count; ++i) {
        uint8_t grb[3] = { pixels[i].g, pixels[i].r, pixels[i].b };

        for (size_t byte_index = 0; byte_index < 3U; ++byte_index) {
            for (int bit_index = 7; bit_index >= 0; --bit_index) {
                send_ws2812_bit((grb[byte_index] >> bit_index) & 0x01U);
            }
        }
    }

    gpio_set_low();
    exit_timing_critical_section();
    delay_us(80U);                  // reset/latch，这一段不再需要纳秒级精度
}
```

注意点：

- CPU 频率变化会改变 cycle 与时间的换算，必须按实际时钟校准。
- GPIO 函数调用本身有开销，不能只把理论延时值直接带入。
- 禁止中断的时间约为 `LED 数量 * 24 * 1.25 us`；100 颗灯已约需 `3 ms`，会影响 Wi-Fi、蓝牙和实时任务。
- 定时器中断也能分段翻转 GPIO，但在 800 kbit/s 级别上中断频率和抖动都较大，不如 RMT 合适。

因此，该方式适合少量灯珠、学习时序或引导阶段；对于运行 FreeRTOS 且有无线业务的 ESP32-S3 应用，优先考虑 RMT 或 SPI。

### 8.2 脉冲外设与 RMT 驱动

RMT（Remote Control Transceiver）可把一个符号直接描述为“电平 0 持续多少 tick，电平 1 持续多少 tick”。这与 WS2812B 的“高电平宽度决定 0/1”天然匹配。

本项目的 RMT 分支就是走这条路线。`components/RGB_LED/ws2812_led_strip.c` 中将 RMT 分辨率设为 `10 MHz`，即 `1 tick = 0.1 us`。随仓库维护的 `led_strip` RMT encoder 对 `LED_MODEL_WS2812` 使用的时序是：

```text
WS2812B 0 -> RMT symbol { high 3 ticks, low 9 ticks }   // 0.3 us + 0.9 us
WS2812B 1 -> RMT symbol { high 9 ticks, low 3 ticks }   // 0.9 us + 0.3 us
reset      -> RMT symbol { low 280 us 左右 }
```

对应到本项目，业务代码不手动生成 `rmt_symbol_word_t` 数组，而是把 GPIO、灯数、GRB 格式和 RMT 参数交给 `led_strip_new_rmt_device()`。后续设置颜色时调用 `led_strip_set_pixel()` 写入组件内部像素缓冲，再调用 `led_strip_refresh()` 触发 RMT 输出。

项目代码逻辑可以按下面这样理解：

```c
// components/RGB_LED/ws2812_led_strip.c
static led_strip_handle_t led_strip;

void ws2812_rmt_led_init(void)
{
    led_strip_config_t strip_config = {
        .strip_gpio_num = RGB_LED_GPIO_PIN,          // 当前为 GPIO_NUM_48
        .max_leds = RMT_AND_SPI_STRIP_LED_NUMBERS,   // 当前为 1 颗
        .led_pixel_format = LED_PIXEL_FORMAT_GRB,
        .flags.invert_out = false,
    };

    led_strip_rmt_config_t rmt_config = {
        .resolution_hz = 10 * 1000 * 1000,            // 10 MHz, 1 tick = 0.1 us
        .flags.with_dma = false,
        .mem_block_symbols = 64,
    };

    ESP_ERROR_CHECK(led_strip_new_rmt_device(&strip_config, &rmt_config, &led_strip));
    ESP_ERROR_CHECK(led_strip_clear(led_strip));
}

static void ws2812_set_first_pixel(uint8_t red, uint8_t green, uint8_t blue)
{
    uint8_t scaled_red = ws2812_apply_brightness(red);
    uint8_t scaled_green = ws2812_apply_brightness(green);
    uint8_t scaled_blue = ws2812_apply_brightness(blue);

    ESP_ERROR_CHECK(led_strip_set_pixel(led_strip, 0,
                                        scaled_red,
                                        scaled_green,
                                        scaled_blue));
    ESP_ERROR_CHECK(led_strip_refresh(led_strip));
}
```

`led_strip_set_pixel()` 的接口参数仍然是 `red, green, blue`。RMT 后端内部会按 WS2812B 要求写成 `GRB` 缓冲：

```c
// components/espressif__led_strip/src/led_strip_rmt_dev.c
rmt_strip->pixel_buf[start + 0] = green & 0xFF;
rmt_strip->pixel_buf[start + 1] = red & 0xFF;
rmt_strip->pixel_buf[start + 2] = blue & 0xFF;
```

`led_strip_refresh()` 最终会把这段 GRB 缓冲交给 RMT encoder：

```c
rmt_transmit(rmt_strip->rmt_chan,
             rmt_strip->strip_encoder,
             rmt_strip->pixel_buf,
             rmt_strip->strip_len * rmt_strip->bytes_per_pixel,
             &tx_conf);
rmt_tx_wait_all_done(rmt_strip->rmt_chan, -1);
```

因此，本项目 RMT 路径的数据流是：

```text
ws2812_rmt_led_init()
  -> led_strip_new_rmt_device()
  -> led_strip_set_pixel(red, green, blue)
  -> RMT 后端写入 GRB pixel_buf
  -> led_strip_refresh()
  -> rmt_transmit()
  -> RMT encoder 输出 WS2812B 波形和 reset 低电平
```

RMT 方式的特点是：

- 波形由硬件按 tick 输出，CPU 不需要在每个高低电平边沿忙等。
- 波形启动后不容易受 FreeRTOS 调度影响。
- 灯珠较多时要关注 RMT symbol 内存、编码器持续供数和 DMA 配置，否则频繁中断仍会增加 CPU 压力。

ESP-IDF 的 `espressif/led_strip` 组件已经封装了 RMT 后端，本项目也是直接使用 `led_strip_new_rmt_device()`，没有在应用层手动拼 RMT symbol。

### 8.3 SPI 原理与 `led_strip` SPI 驱动

WS2812B 不是 SPI 设备，但 ESP32-S3 的 SPI MOSI 可以被当作定时稳定的单线波形输出。做法是把每个 WS2812B bit 扩展为多个 SPI bit：

```text
WS2812B bit 0 -> SPI bits 100 -> 高 1 拍，低 2 拍
WS2812B bit 1 -> SPI bits 110 -> 高 2 拍，低 1 拍
```

本项目随仓库维护的 `led_strip` SPI 后端默认 SPI 时钟是 `2.5 MHz`，每拍约 `0.4 us`，3 拍约 `1.2 us`。于是 MOSI 上的 `100` 近似“高短低长”，`110` 近似“高长低短”。不同灯珠容差和驱动编码可能选用不同的扩展比和 SPI 时钟，关键是最终 MOSI 波形落在 WS2812B 的时序容差内。

数据流为：

```text
RGB 颜色 -> GRB 24 bit -> SPI 波形编码 -> DMA/SPI -> MOSI -> WS2812B DIN
```

ESP-IDF 的 `espressif/led_strip` 组件会在内部完成 GRB 排列和 SPI 波形编码。本项目的 `led_strip` SPI 分支对应 `ws2812_spi_led_init()`，代码逻辑如下：

```c
// components/RGB_LED/ws2812_led_strip.c
void ws2812_spi_led_init(void)
{
    led_strip_config_t strip_config = {
        .strip_gpio_num = RGB_LED_GPIO_PIN,          // 当前为 GPIO_NUM_48
        .max_leds = RMT_AND_SPI_STRIP_LED_NUMBERS,   // 当前为 1 颗
        .led_pixel_format = LED_PIXEL_FORMAT_GRB,
        .led_model = LED_MODEL_WS2812,
        .flags.invert_out = false,
    };

    led_strip_spi_config_t spi_config = {
        .clk_src = SPI_CLK_SRC_DEFAULT,
        .flags.with_dma = true,
        .spi_bus = SPI3_HOST,
    };

    ESP_ERROR_CHECK(led_strip_new_spi_device(&strip_config, &spi_config, &led_strip));
    ESP_ERROR_CHECK(led_strip_clear(led_strip));
}

static void ws2812_set_first_pixel(uint8_t red, uint8_t green, uint8_t blue)
{
    uint8_t scaled_red = ws2812_apply_brightness(red);
    uint8_t scaled_green = ws2812_apply_brightness(green);
    uint8_t scaled_blue = ws2812_apply_brightness(blue);

    ESP_ERROR_CHECK(led_strip_set_pixel(led_strip, 0,
                                        scaled_red,
                                        scaled_green,
                                        scaled_blue));
    ESP_ERROR_CHECK(led_strip_refresh(led_strip));
}
```

SPI 后端内部只配置 MOSI，SCLK、MISO、CS 都不输出到外部：

```c
// components/espressif__led_strip/src/led_strip_spi_dev.c
spi_bus_config_t spi_bus_cfg = {
    .mosi_io_num = led_config->strip_gpio_num,
    .miso_io_num = -1,
    .sclk_io_num = -1,
    .quadwp_io_num = -1,
    .quadhd_io_num = -1,
    .max_transfer_sz = led_config->max_leds * bytes_per_pixel * SPI_BYTES_PER_COLOR_BYTE,
};
```

组件内部用 3 个 SPI bit 表示 1 个 WS2812B bit：

```c
#define LED_STRIP_SPI_DEFAULT_RESOLUTION (2.5 * 1000 * 1000)
#define SPI_BYTES_PER_COLOR_BYTE 3

// 颜色 bit 0 -> SPI 100
// 颜色 bit 1 -> SPI 110
static void __led_strip_spi_bit(uint8_t data, uint8_t *buf)
{
    // Each color of 1 bit is represented by 3 bits of SPI,
    // low_level:100, high_level:110.
}
```

设置像素时，SPI 后端同样把 RGB 接口参数写成 GRB 顺序，并且每个颜色字节扩展为 3 字节 SPI 数据：

```c
uint32_t start = index * spi_strip->bytes_per_pixel * SPI_BYTES_PER_COLOR_BYTE;

__led_strip_spi_bit(green, &spi_strip->pixel_buf[start]);
__led_strip_spi_bit(red,   &spi_strip->pixel_buf[start + SPI_BYTES_PER_COLOR_BYTE]);
__led_strip_spi_bit(blue,  &spi_strip->pixel_buf[start + SPI_BYTES_PER_COLOR_BYTE * 2]);
```

刷新时发送整段已经编码好的 SPI 缓冲：

```c
tx_conf.length = spi_strip->strip_len * spi_strip->bytes_per_pixel * SPI_BITS_PER_COLOR_BYTE;
tx_conf.tx_buffer = spi_strip->pixel_buf;
spi_device_transmit(spi_strip->spi_device, &tx_conf);
```

对当前项目的 `1` 颗 GRB 灯来说，刷新长度是：

```text
1 颗灯 * 3 个颜色字节 * 24 个 SPI bit = 72 SPI bit = 9 byte
```

因此，本项目 `led_strip` SPI 路径的数据流是：

```text
ws2812_spi_led_init()
  -> led_strip_new_spi_device()
  -> led_strip_set_pixel(red, green, blue)
  -> SPI 后端按 GRB 顺序编码 pixel_buf
  -> 每个 WS2812B bit 扩展为 3 个 SPI bit
  -> led_strip_refresh()
  -> spi_device_transmit()
  -> SPI3 MOSI 输出到 WS2812B DIN
```

尤其要注意：

- SPI 后端虽然只使用 MOSI，但会占用整条 SPI bus；WS2812B 没有 CS，因此不能把这条总线再分享给 LCD、Flash 或其他 SPI 设备。
- `led_strip_set_pixel()` 只更新内存中的颜色，`led_strip_refresh()` 才会把整条灯带数据输出。
- SPI + DMA 适合连续输出较长灯带，代价是一条独占的 SPI bus 和编码后的缓冲区。

## 9. SPI 模拟方式的数据膨胀

如果采用 `0 -> 100`、`1 -> 110` 的方式编码，那么 1 个 WS2812B bit 会变成 3 个 SPI bit。

一颗灯原始数据：

```text
24 bit = 3 byte
```

编码后：

```text
24 bit * 3 = 72 SPI bit = 9 byte
```

也就是说：

```text
1 颗灯原始颜色数据：3 字节
1 颗灯 SPI 波形数据：9 字节
```

例如颜色字节 `0xA5`：

```text
原始 bit:
10100101

扩展后:
110 100 110 100 100 110 100 110
```

这 24 个 SPI bit 从 MOSI 发出去后，WS2812B 不知道它们是 SPI，它只看到一串高低电平宽度合适的波形。

## 10. 为什么 WS2812B 对时序敏感

WS2812B 没有单独时钟线，所以它只能根据 DIN 的高低电平宽度判断 bit 值。

如果时序不准，就可能出现：

- 高电平太短，`1` 被误判成 `0`。
- 高电平太长，`0` 被误判成 `1`。
- bit 周期不稳定，后续数据整体错位。
- reset 时间不足，数据没有正确锁存。
- 中途被中断打断，整条灯带显示异常。

因此驱动 WS2812B 时，最重要的不是“把字节发出去”，而是“让每一个 bit 的波形持续时间稳定”。

## 11. 常见排查思路

如果 WS2812B 不亮或颜色异常，可以按数据流反向排查：

| 现象 | 可能原因 | 排查重点 |
| --- | --- | --- |
| 完全不亮 | 供电、地线、DIN 接错、reset/波形没有输出 | 电源、电平、GPIO、示波器波形 |
| 颜色顺序不对 | RGB/GRB 顺序错误 | 检查颜色通道排列 |
| 只有第一颗正常 | 级联方向、DOUT/DIN、数据长度问题 | 检查灯带方向和发送字节数 |
| 随机闪烁 | 时序不稳定、中断影响、供电不足 | 检查波形宽度和电源纹波 |
| 后面灯颜色错位 | 少发/多发 bit，reset 不足 | 检查整帧长度和 latch 时间 |
| 亮度太高或太暗 | 颜色值或全局亮度缩放 | 检查 RGB 数值和限流设计 |

重点记住：颜色异常不一定是颜色计算错，也可能是前面的某个 bit 时序错了，导致后续数据整体错位。

## 12. 关键结论

WS2812B 的底层驱动可以用一条数据流概括：

```text
应用层 RGB 颜色
  -> 可选亮度缩放
  -> 转成 WS2812B 常用的 GRB 顺序
  -> 按高位在前展开成 bit 流
  -> 每个 bit 编码为高低电平持续时间
  -> DIN 输出连续脉冲
  -> 每颗灯按顺序消费 24 bit
  -> reset 低电平锁存
  -> 内部 PWM 显示颜色
```

理解 WS2812B 时，最重要的是把“数据”和“波形”分开看：

- 数据层：每颗灯 24 bit，常见顺序是 `G R B`。
- 协议层：`0` 和 `1` 由高低电平持续时间区分。
- 传输层：主控只负责输出一根 DIN 波形，没有地址、寄存器和 ACK。
- 级联层：每颗灯消费自己的 24 bit，把剩余数据传给下一颗。
- 显示层：reset 后锁存数据，内部 PWM 根据颜色值控制亮度。

所以，驱动 WS2812B 的本质就是：**把一串 RGB 颜色数组，转换成严格定时的单线脉冲流。**
